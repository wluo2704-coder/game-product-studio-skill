# Role selection rules

Start with the Discovery Lead. Add a role only for independently testable work with a stable handoff; the Lead remains sole integrator.

| Signal | Mode / role |
| --- | --- |
| Idea is still vague | Discovery cell; optional read-only Loop Analyst or Feature Critic gives the Lead notes, never user questions. |
| One small loop and a prototype decision | Prototype mode: Lead, bounded Prototype Architect input, QA, Reflection. |
| A confirmed game-internal Agent changes that loop | Agent-MVP mode adds Product-Agent Designer. It does not authorize keys, model choice, backend, or durable task creation. |
| Several systems or sustained content/art work | Standard: add design and art with bounded inbound directories. |
| Multi-platform, live/networked, complex 3D | Studio: add technical, audio, and release specialists only for affected scope. |
| Existing project lacks ownership/evidence | Takeover: Lead, QA, Reflection audit before new implementation. |

## Boundaries

- Lead: the only user-facing voice and sole writer of source/configuration/integration/Candidate/formal delivery.
- Contributors: write only their named inbound artifacts; never alter integrated game files.
- QA: read-only and independently evidence-based. For in-scope direct-scene tests, QA may invoke only the Lead-provided, candidate-bound GM allowlist and must log/reset every run.
- Reflection: read-only and independently evidence-based; it does not receive GM control unless a separate scope explicitly requires observation-only access.
- Long-lived task creation, dependency installation, credentials, publishing, and boundary expansion require explicit user authorization.
